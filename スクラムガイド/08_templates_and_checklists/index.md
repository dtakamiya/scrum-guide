# 8. 実践テンプレート・チェックリスト集

アジャイル開発の各シーンで活用できるテンプレートとチェックリストをまとめました。
これらはあくまで基本的な型です。自身のチームやプロジェクトの状況に合わせて自由にカスタマイズしてください。

---

## 1. プロジェクト計画・戦略系

プロジェクトの立ち上げや全体像の共有に役立つテンプレートです。

### 1.1 プロジェクト計画テンプレート
アジャイルなプロジェクト全体の計画を立てるための基本的なフレームワークです。ロードマップ、リソース、リスクなどを管理します。
- [ ] プロジェクトビジョン・目標
- [ ] 主要なマイルストーン
- [ ] リリース計画
- [ ] チーム体制
- [ ] 想定されるリスクと対策

### 1.2 プロダクトロードマップ
プロダクトが将来的にどのように進化していくかを視覚的に示したものです。チームとステークホルダーの認識を合わせます。
- **期間**: [例: 3ヶ月、6ヶ月、1年]
- **テーマ/目標**: [例: Q1 ユーザー獲得強化, Q2 支払い機能改善]
- **主要な機能/エピック**: [上記テーマに紐づく具体的な機能]
- **ステータス**: [計画中, 進行中, 完了]

### 1.3 インセプションデッキ
プロジェクトの全体像を10の質問に答える形式で明確にするためのテンプレートです。
- [ ] 我々は何故ここにいるのか？
- [ ] エレベーターピッチ
- [ ] パッケージデザイン
- [ ] やらないことリスト
- [ ] 「ご近所さん」を探せ
- [ ] 解決策を描く
- [ ] 夜も眠れない問題
- [ ] 期間を見極める
- [ ] 何を諦めるか
- [ ] 何がどれだけ必要か

## 2. バックログ管理系

プロダクトバックログとスプリントバックログを効果的に管理するためのテンプレートです。

### 2.1 プロダクトバックログ
開発する機能や要件を優先順位順にリスト化したものです。

#### バックログアイテムの例
- **機能**: ユーザーに新しい価値を提供するもの (例: `写真への「いいね」機能`)
- **バグ修正**: 既存機能の不具合修正 (例: `ログイン画面でパスワードが保存されない`)
- **技術的負債の返済**: 将来の開発効率を上げるための内部的な改善 (例: `非推奨ライブラリのアップデート`)
- **調査・知識獲得**: 不確実性を減らすための調査 (例: `新しい決済APIの実現可能性調査`)

#### 作成の4ステップ
1.  **プロダクトゴールを決める**: プロダクトのビジョンやミッションを明確にする。
2.  **アイテムを作成し、追加する**: ゴールを実現するための要求（ユーザーストーリーなど）をリストアップする。
3.  **アイテムに優先順位をつける**: ビジネス価値と開発コストを考慮してアイテムを並び替える。（MoSCoWメソッド, 狩野モデル, 価値-コストマトリクスなどの手法が有効）
4.  **優先順位の高いアイテムを詳細化する**: すぐに開発に着手できるよう、ユーザーストーリーや受け入れ基準を明確にする。

- **優先度**: (High, Mid, Low など)
- **見積もり (ポイント)**:
- **ステータス**: (Todo, In Progress, Done)
- **担当スプリント**:
- **備考**:

特定のスプリントで達成すべきプロダクトバックログアイテムと、それを達成するための具体的なタスクリストです。
- **スプリントゴール**:
- **選択されたプロダクトバックログアイテム**:
- **タスクリスト**:
  - **タスクID**:
  - **タスク名**:
  - **担当者**:
  - **見積もり (時間)**:
  - **ステータス**: (Todo, In Progress, Done)
- **バーンダウンチャート**:

### 2.3 バックログリファインメント チェックリスト
バックログを健全な状態に保つための定期的な見直し活動です。
- [ ] プロダクトバックログアイテムはプロダクトゴールに沿っているか？
- [ ] 優先順位は最新のビジネス状況を反映しているか？
- [ ] 優先度の高いアイテムは、開発チームが着手できるレベルまで詳細化されているか？ (INVEST原則)
- [ ] 見積もりは妥当か？大きすぎるアイテムは分割されているか？
- [ ] 不要になったアイテムは削除されているか？

## 3. スプリントイベント系

スクラムの各イベントを円滑に進めるためのテンプレートとチェックリストです。

### 3.1 スプリントプランニング
- **議題**:
  - [ ] スプリントゴールの設定
  - [ ] 対象となるプロダクトバックログアイテムの選択
  - [ ] タスクへの分割と見積もり
  - [ ] スプリントバックログの作成
- **準備**:
  - [ ] プロダクトオーナーは優先順位付けされたバックログを準備したか？
  - [ ] チームのベロシティ（実績）は明確か？
- **成果物**:
  - [ ] スプリントゴール
  - [ ] スプリントバックログ

### 3.2 デイリースタンドアップ
- **3つの質問**:
  - 昨日やったことは何か？
  - 今日やることは何か？
  - 何か困っていること（障害）はあるか？
- **目的**: 1日の計画を立て、チームの連携を促進すること。問題解決の場ではない。

### 3.3 スプリントレビュー
- **議題**:
  - [ ] スプリントで完成したインクリメントのデモ
  - [ ] ステークホルダーからのフィードバック収集
  - [ ] プロダクトバックログの見直しと今後の見通しの共有
- **準備**:
  - [ ] デモの準備は完了しているか？
  - [ ] 主要なステークホルダーは招待されているか？

### 3.4 レトロスペクティブ（ふりかえり）
チームのプロセスを改善するための話し合いです。様々な手法があります。

#### KPT (Keep, Problem, Try)
- **Keep**: 良かったこと、続けたいこと
- **Problem**: 問題だったこと、改善したいこと
- **Try**: 次に試すこと (具体的なアクション)

#### Starfish (ヒトデ)
- **Start Doing**: 新しく始めること
- **Stop Doing**: やめること
- **Keep Doing**: 続けること
- **More of**: もっとやること
- **Less of**: 減らすこと

## 4. 要求定義・仕様系

要求を明確にし、チーム内外での認識を合わせるためのテンプレートです。

### 4.1 ユーザーストーリー
ユーザーにとっての価値を簡潔に表現するためのフォーマットです。

#### テンプレート
`「<ペルソナ>として、<価値>のために<機能>が欲しい」`

*   **ペルソナ (As a <Persona>)**: この機能を誰が使うのか？ (例: `旅行サイトの会員`)
*   **機能 (I want to <Want>)**: 何をしたいのか？ (例: `過去の旅行履歴を見たい`)
*   **価値 (So that <Value>)**: それによって何が嬉しいのか？ (例: `同じホテルを簡単に再予約するため`)

#### 良いユーザーストーリーの指針 (INVEST)
- [ ] **I - Independent (独立している)**: 他のストーリーに依存していないか？
- [ ] **N - Negotiable (交渉可能である)**: 詳細はチームで対話して決められる余地があるか？
- [ ] **V - Valuable (価値がある)**: ユーザーにとって明確な価値があるか？
- [ ] **E - Estimable (見積可能である)**: チームが見積もれる大きさか？
- [ ] **S - Small (小さい)**: 1スプリントで完了できる大きさか？
- [ ] **T - Testable (テスト可能である)**: 完了したことを確認できるか？

#### 具体例
- **良い例**: `旅行サイトの会員として、同じホテルを簡単に再予約するために、過去の旅行履歴を見たい。`
- **悪い例**: `ユーザーテーブルに最終予約日カラムを追加する。` (←これはユーザーストーリーではなくタスク)

### 4.2 受け入れ基準 (Acceptance Criteria)
ユーザーストーリーが「完成」したと判断するための具体的な条件リストです。
`Given (前提) - When (操作) - Then (結果)` の形式で書くと明確になります。

**ユーザーストーリー**: `旅行サイトの会員として、パスワードを忘れた時に再設定できるようにしたい。`
- **受け入れ基準1**:
  - **Given**: ログイン画面を開いている
  - **When**: 「パスワードを忘れた場合」リンクをクリックする
  - **Then**: パスワード再設定ページに遷移する
- **受け入れ基準2**:
  - **Given**: パスワード再設定ページで登録済みのメールアドレスを入力している
  - **When**: 「再設定メールを送信」ボタンをクリックする
  - **Then**: 「再設定用のメールを送信しました」と表示され、実際にメールが届く

### 4.3 Definition of Done (DoD: 完成の定義)
すべてのユーザーストーリーが共通で満たすべき品質基準のチェックリストです。チームで合意して作成します。
- [ ] コードレビューが完了している
- [ ] ユニットテストが書かれ、すべてパスしている
- [ ] 受け入れ基準がすべて満たされている
- [ ] QA担当者によるテストが完了している
- [ ] ドキュメントが更新されている
- [ ] 本番環境にデプロイ可能である

### 4.4 Definition of Ready (DoR: 準備完了の定義)
プロダクトバックログアイテムがスプリントに着手できる状態であると判断するためのチェックリストです。
- [ ] ユーザーストーリーが書かれている
- [ ] 受け入れ基準が明確になっている
- [ ] 依存関係が解決されている
- [ ] 開発チームによって見積もりが完了している
- [ ] プロダクトオーナーによって優先順位が付けられている

## 5. パートナー連携系

複数のベンダーやパートナーと協働する際のテンプレートです。

### 5.1 One Team 構築チェックリスト
パートナー企業と一体感のあるチームを構築するためのチェックリストです。
- [ ] 共通のプロダクトビジョンが共有されているか？
- [ ] チームメンバー全員が同じ場所（物理的または仮想的）で作業しているか？
- [ ] 役割の境界線が曖昧で、全員がプロダクトの成功に責任を持っているか？
- [ ] 定期的なチームビルディング活動が行われているか？
- [ ] コミュニケーションの障壁（言語、文化、所属企業）が取り除かれているか？

### 5.2 契約モデル評価シート
アジャイル開発に適した契約形態を選択するための評価シートです。

#### 請負契約の評価
- [ ] 仕様が明確で変更が少ないか？
- [ ] リスクが低く、予測可能なプロジェクトか？
- [ ] 発注者が詳細な要件を定義できるか？
- [ ] 受注者が固定価格でリスクを負えるか？

#### 準委任契約の評価
- [ ] 要件が不明確で変化が予想されるか？
- [ ] アジャイル開発を採用しているか？
- [ ] パートナーシップを重視しているか？
- [ ] リスクを共有する意思があるか？

### 5.3 パートナー連携改善アクションシート
レトロスペクティブでパートナー連携の改善点を特定し、アクションに繋げるためのシートです。

#### 改善対象エリア
- **コミュニケーション**: [ ] 頻度、[ ] 方法、[ ] 内容
- **役割分担**: [ ] 責任の明確化、[ ] 権限の委譲、[ ] 意思決定プロセス
- **技術的協力**: [ ] 知識共有、[ ] コードレビュー、[ ] ペアプログラミング
- **プロセス統合**: [ ] 開発フロー、[ ] 品質基準、[ ] リリースプロセス

#### アクション項目
- **アクション**: [具体的な改善行動]
- **担当者**: [責任者]
- **期限**: [完了予定日]
- **成功基準**: [どうなれば成功か？]

## 6. メトリクス・KPI系

チームの成長と成果を可視化するためのテンプレートです。

### 6.1 GQM（ゴール・質問・指標）設計シート
意味のあるメトリクスを設計するための思考フレームワークです。

#### Step 1: ゴールの設定
- **ビジネスゴール**: [例: 顧客満足度を20%向上させる]
- **対象**: [例: モバイルアプリのユーザー]
- **期間**: [例: 3ヶ月]
- **成功基準**: [例: NPSスコアが50以上]

#### Step 2: 質問の設定
- **質問1**: [例: ユーザーはアプリをどの程度活用しているか？]
- **質問2**: [例: ユーザーはどの機能を最も使用しているか？]
- **質問3**: [例: ユーザーはアプリにどの程度満足しているか？]

#### Step 3: 指標の設定
- **指標1**: [例: 日次アクティブユーザー数]
- **指標2**: [例: 機能別使用率]
- **指標3**: [例: アプリ内評価スコア]

### 6.2 心理的安全性測定シート
チームの心理的安全性を定期的に測定し、改善するためのシートです。

#### 測定項目（1-5段階評価）
- [ ] チーム内で間違いを指摘しても安全だ
- [ ] チームメンバーは困難な問題や課題について率直に話し合える
- [ ] チーム内で異なる意見を言っても安全だ
- [ ] チーム内でリスクを取っても安全だ
- [ ] チームメンバーに助けを求めることは難しい

#### 改善アクション
- **低評価項目**: [改善が必要な項目]
- **改善アクション**: [具体的な行動]
- **担当者**: [責任者]
- **期限**: [完了予定日]

### 6.3 ダッシュボード設計チェックリスト
効果的なダッシュボードを設計するためのチェックリストです。
- [ ] ダッシュボードの目的が明確か？
- [ ] 対象ユーザー（チーム、マネージャー、ステークホルダー）が明確か？
- [ ] 表示するメトリクスは意味があり、行動に繋がるか？
- [ ] データの更新頻度は適切か？
- [ ] 視覚的に理解しやすいか？
- [ ] コンテキスト（背景情報）が含まれているか？
- [ ] 異常値やトレンドが分かりやすいか？

## 7. トラブルシューティング系

よくある問題の診断と解決のためのテンプレートです。

### 7.1 スクラム導入問題診断シート
スクラムがうまく機能していない場合の原因を特定するための診断シートです。

#### 組織レベルの問題
- [ ] 経営陣のアジャイル理解が不足している
- [ ] 従来型の管理手法が残っている
- [ ] チームの自律性が認められていない
- [ ] 短期的な成果を重視しすぎている

#### チームレベルの問題
- [ ] スクラムイベントが形式的になっている
- [ ] バックログが適切に管理されていない
- [ ] チームメンバーの役割が曖昧
- [ ] 心理的安全性が確保されていない

#### プロセスレベルの問題
- [ ] スプリントのリズムが乱れている
- [ ] インクリメントが完成していない
- [ ] 振り返りから改善アクションが出ない
- [ ] 外部からの干渉が多い

### 7.2 アンチパターン対策シート
よくあるアンチパターンを特定し、対策を立案するためのシートです。

#### ダークアジャイル対策
- **症状**: [例: ベロシティをノルマとして課している]
- **根本原因**: [例: 従来型の管理思考が残っている]
- **対策**: [例: アジャイルの価値観を再学習する]
- **担当者**: [例: スクラムマスター]
- **期限**: [例: 1ヶ月以内]

#### 形骸化アジャイル対策
- **症状**: [例: デイリースクラムが進捗報告会になっている]
- **根本原因**: [例: イベントの目的が理解されていない]
- **対策**: [例: 各イベントの目的を再確認する]
- **担当者**: [例: チーム全員]
- **期限**: [例: 次のスプリントから]

### 7.3 障害管理シート
チームの進行を妨げる障害を体系的に管理するためのシートです。

## 使用方法

1. **状況に応じた選択**: プロジェクトの状況やチームの成熟度に応じて、適切なテンプレートを選択してください。
2. **カスタマイズ**: これらのテンプレートは基本的な型です。チームの状況に合わせて自由にカスタマイズしてください。
3. **継続的改善**: 使用したテンプレートの効果を定期的に評価し、改善していってください。
4. **共有・学習**: 効果的なテンプレートは他のチームと共有し、組織全体の学習に活用してください。
